I previously published “Things I’ve learned about sending email”. A lot of people seemed to like the post and it was a great intro to sending email or quick cheat sheet.

Some people followed up with me and wanted to know more about the building and testing process when it comes to HTML email. Here’s a brain dump of what I’ve learned about actually building the emails, writing the HTML/CSS and so forth.

Email clients and rendering engines

Email clients use different rendering engines to render HTML emails:

  • Apple Mail, Outlook for Mac, Android Mail and iOS Mail use WebKit
  • Outlook 2000/02/03 use Internet Explorer
  • Outlook 2007/10/13 use Microsoft Word (yes, Word!)
  • Web clients use their browser’s respective engine e.g. Safari uses WebKit, Chrome uses Blink

Gmail strips out <link> tags and CSS in the <style> tags and any CSS that isn’t inlined. Not just Gmail web but also the native Gmail mobile apps.

Clients will also add their own styles on top of yours e.g. Gmail sets all <td> font to font-family: arial,sans-serif;

Litmus has a great website dedicated to the current email client market share, based on their own internal stats (which is millions of emails).

As of April 2015 the top 5 looks like so:

  1. Apple iPhone 28%
  2. Gmail 16%
  3. Apple iPad 11%
  4. Outlook 8%
  5. Apple Mail 8%

Of course you should look at your own stats to know which clients you need to focus on for your recipients.

Multi-part and plain text

When you send an email it’s important to send both plain text and html. You do this by sending your email as Multi-part MIME. Even in a world where you think every one uses HTML clients.

Most ESPs will construct the MIME for you so you don’t really need to worry about it. Some will also create a plain text version for you based on your HTML version.

Note that with the recent launch of Apple Watch, there is a new MIME part available: text/watch-html. This content will only be displayed in the Apple Watch and any other clients that support this MIME type.

Use tables not divs

Divs have positioning and box model issues in different clients, in particular those that use Word to render. You can use divs if you want but best to stick to tables.

  • Tables over divs

  • ffffff over #fff

  • Padding over margins
  • CSS2 over CSS3
  • HTML4 over HTML5
  • background-color over background
  • HTML attributes over CSS
  • Inline CSS over CSS in <style> tags

Of course these approaches are recommended best practices. You can certainly go beyond this.

When using tables don’t forget border="0" cellpadding="0" cellspacing="0". If you’re using Premailer they have special CSS declarations for applying these HTML attributes.

Inline CSS

Gmail will strip any CSS that isn’t inlined. You have a couple of options here:

  • Write CSS inline as you go
  • Use a web-based CSS inliner
  • Use a programmatic CSS inliner
  • Let your ESP handle the inlining for you (if they support it)

Writing inline isn’t exactly a scalable or maintainable solution so I tend not to, but I know a lot of email builders prefer control over it this way. If you do write your CSS inline manually then I recommend making use of snippets and/or a templating language with partials and helpers. This way you don’t have to repeat yourself.

Web based inliners include Campaign Monitor, Mailchimp and Zurb’s Ink.

For a programmatic inliner I typically use the Premailer gem, or another package based on it.

ESPs that support inlining include Campaign Monitor and Mailchimp.

Conditional targeting

We can apply specific CSS styles and show/hide elements for different versions of Outlook.

This would target all Microsoft Word based versions of Outlook:

<!--[if mso]>
MARKUP GOES HERE
<![endif]-->

This would target all IE based versions of Outlook:

<!--[if (IE)]>
MARKUP GOES HERE
<![endif]-->

We can also target specific versions of Outlook:

<!--[if mso 9]>
MARKUP GOES HERE
<![endif]-->

More on conditional statements for Outlook.

We can target WebKit based clients with a media query like this:

@media screen and (-webkit-min-device-pixel-ratio:0) {  
/* CSS GOES HERE */
}

Type

In general you’re best to stick with standard fonts. That includes Arial etc.

We can use other fonts like Google fonts. You have to put them behind a webkit conditional otherwise Outlook will fuck up.

Paul Airey has a great article here and Jason Rodriguez published the chapter of his book on type.
